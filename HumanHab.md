RESUMEN CONSOLIDADO DEL PROYECTO
Sistema Adaptativo de Hábitos, Productividad y Regulación
1️⃣ VISIÓN GENERAL
Estamos diseñando un sistema que:
Ayuda a construir hábitos desde cero o mejorarlos.
Aumenta productividad sin sacrificar salud mental.
Reduce estrés y fricción.
Mide recuperación más que perfección.
Aprende del usuario.
Interviene con empatía.
Se adapta al contexto diario.
No es:
Un habit tracker clásico.
Una app de frases motivacionales.
Una app de meditación.
Una herramienta rígida.
Es un entrenador adaptativo de comportamiento y regulación personal.
2️⃣ PRINCIPIO CENTRAL
La transformación no depende de no fallar.
Depende de qué tan rápido volvés después de fallar.
Por eso, la métrica clave es:
🔁 Recovery Speed
Tiempo que tarda el usuario en retomar un hábito tras una interrupción.
La app no castiga la caída.
Optimiza la recuperación.
3️⃣ FILOSOFÍA DEL SISTEMA
Estabilidad antes que crecimiento.
Empatía antes que exigencia.
Sugerir, no imponer.
Menos fricción antes que más control.
Adaptarse a la persona, no al revés.
No comparar con ideales irreales.
4️⃣ TIPO DE USUARIO
El sistema sirve tanto para:
Personas sin hábitos.
Personas inconsistentes.
Personas productivas pero sobrecargadas.
Emprendedores de alto rendimiento.
Personas que quieren salir de un estado actual limitado.
No es exclusivo para un perfil.
Por eso existe la entrevista inicial.

Capa 1 – Reglas duras (protección)

Capa 3 – IA interpretativa

9️⃣ BASELINE DINÁMICO
El usuario se evalúa contra:
Su propio promedio real.
No contra ideales.
No contra otros.
No contra estándares irreales.
🔟 CAPACIDAD DIARIA
Cada día se clasifica según su capacidad:
Empujar

sobreestimulación

Reducir carga mental
Transformar tareas dispersas en hábitos sostenibles
Permitir integrar acciones importantes sin rigidez

El cambio depende del usuario.
El sistema acompaña.
🧱 Diseño alineado al cerebro humano
Acepta fatiga.
Acepta dopamina instantánea.
Acepta variabilidad emocional.
No lucha contra la biología.

🧱 Empatía estructural
La empatía no es tono, es arquitectura.
Nunca castiga.

Nunca usa culpa como motor.

🧱 Centralidad absoluta del usuario
Toda decisión se evalúa con:
¿Esto realmente ayuda al usuario en su vida real?
Si la respuesta es dudosa, se descarta.
🧱 Impacto real
El sistema no busca entretenimiento ni sensación artificial de productividad.
Busca:
Mejorar relación con el tiempo
Reducir estrés
Sostener procesos sin romperse

Definición de límites explícitos.

No garantiza cambios.
No ofrece plazos mágicos.
No vende transformaciones rápidas.

Nunca usa:

presión

No vende datos.

4️⃣ OBJETIVO CENTRAL
Ayudar al usuario a construir y sostener hábitos en el tiempo, incluso en contextos variables, desarrollando una relación más sana con el proceso, reduciendo estrés y fortaleciendo su identidad personal.
Este objetivo:
Es núcleo, no consecuencia.
Explica por qué luego hay más productividad.
Explica por qué hay menos estrés.
Explica por qué mejora la identidad.

5️⃣ DEFINICIÓN DE ÉXITO Y FRACASO
Esta es una de las decisiones más diferenciales.
🟢 ÉXITO
El sistema está funcionando cuando:
El usuario sigue intentando y volviendo, aunque sea de forma imperfecta.
Éxito significa:
Mantener el vínculo con el proceso.
No rendirse consigo mismo.
Volver después de fallar.
No significa:
Cumplir perfecto.
Tener rachas largas.
Ser altamente productivo.
🔴 FRACASO
Fracaso no es:
Fallar un día.
Tener una semana mala.
Reducir ejecución.
Fracaso es:
Abandonar el proceso de forma sostenida y romper el vínculo con la herramienta.
Y aún así:
No hay juicio. Solo se reconoce que ya no está en uso.

BLOQUE 2 – MODELO DE DECISIÓN (Corazón del sistema)
Este bloque define cómo piensa y decide HumanHab.
Es el motor lógico que gobierna:
estados operativos
hábitos
expansión / protección
adaptación en el tiempo
No es UX.
No es narrativa.
Es arquitectura cognitiva.
1️⃣ VARIABLES DEL SISTEMA (Inputs estructurales)
Se definieron 4 variables núcleo:
1. Energía (E)
Capacidad fisiológica y mental disponible.
Determina volumen de acción posible.
2. Estrés (S)
Carga acumulada no procesada.
No es enemigo, pero es limitante si cruza umbral.
3. Ánimo (A)
Estado emocional basal.
Modula fricción, pero no habilita expansión por sí solo.
4. Claridad (C)
Capacidad de comprensión, foco y coherencia.
Se refinó internamente en:
Claridad Cognitiva (CC)
Coherencia Interna (CI)
Externamente sigue siendo una sola variable.
Decisión clave:
Claridad tiene prioridad sobre motivación.
2️⃣ UMBRALES
Se definieron zonas funcionales:
Críticos
Energía muy baja
Estrés muy alto
Claridad muy baja
Funcionales
Zona operativa media.
Óptimos
Zona segura para expansión.
Decisión clave:
El sistema trabaja por zonas de funcionamiento, no por valores aislados.
3️⃣ ESTADOS OPERATIVOS
El sistema siempre asigna uno de tres estados:
🔴 PROTEGER
Objetivo: preservar integridad.
Se reducen decisiones
No se agregan hábitos
No se expande
Se prioriza recuperación
🟡 ESTABILIZAR
Objetivo: regular y ordenar.
Se ajustan hábitos
Se mejora claridad
No se hacen cambios grandes
🟢 EXPANDIR
Objetivo: crecer con criterio.
Se permiten decisiones estratégicas
Puede aumentarse intensidad
Puede agregarse un solo hábito nuevo
Condición:
Energía + Claridad en tendencia positiva y recovery estable.
Decisión clave:
El usuario no elige el estado. El sistema lo asigna.
4️⃣ REGLAS DURAS
Reglas no negociables:
Claridad baja → nunca expandir
Energía crítica → proteger
Estrés alto sostenido → forzar estabilización o protección
Ánimo nunca habilita expansión
Dos variables en riesgo → priorizar protección
Decisión clave:
Las reglas duras evitan autoengaño bajo estrés.
5️⃣ RECOVERY & ADAPTACIÓN
Se definió:
Recovery Speed
Velocidad de retorno a baseline.
Influye en:
futura agresividad
habilitación de expansión
duración en estados protectores
Adaptación del sistema
El sistema aprende:
patrones de recaída
impacto real de hábitos
sostenibilidad personal
tendencia de variables
Ajusta:
timing
intensidad
sugerencias futuras
Decisión clave:
El sistema no premia intensidad, premia sostenibilidad.
6️⃣ LÓGICA DE CHECK-IN
Función: alimentar el modelo de decisión.
Características:
1 check-in diario base
Baja fricción
Subjetivo pero estructurado
Tendencias > momentos aislados
El sistema evalúa:
valor actual
tendencia reciente
cruce de umbrales
coherencia con historial
Output del check-in:
Estado del día
Objetivo operativo
Sugerencias (permitido / no recomendado)
Decisión clave:
Una respuesta aislada no redefine el sistema.
7️⃣ GESTIÓN ADAPTATIVA DE HÁBITOS
El hábito no es una tarea fija.
Es una unidad viva regulada por el estado.
Principio rector:
El hábito se adapta al estado, no el estado al hábito.
Ciclo de vida del hábito:
Ingreso
Observación
Ajuste
Consolidación o pausa
Reglas por estado:
PROTEGER
No agregar hábitos
Reducir intensidad si es necesario
ESTABILIZAR
Ajustar hábitos existentes
Buscar consistencia
EXPANDIR
Aumentar intensidad o
Agregar un solo hábito nuevo
Seguimiento
El sistema evalúa:
cumplimiento
impacto en variables
costo energético
sostenibilidad
Decisión clave:
Cumplir no es suficiente.
Importa el impacto.
8️⃣ LÓGICA DE ESCUCHA Y MEMORIA
El sistema puede recibir input por audio o texto.
Extrae:
intención
hábitos mencionados
fricciones
estado implícito
nivel estimado de conciencia
Lo transforma en memoria estructurada:
hábitos activos
riesgos detectados
carga estimada
tendencias
Decisión clave:
La memoria es operativa, no histórica.
9️⃣ RELACIÓN CON NIVELES
Se definió que:
Existen niveles de autonomía
El sistema modula intervención según nivel
La cantidad de niveles no pertenece a este bloque
BLOQUE 2 no define UX, solo lógica.
🔟 PRINCIPIOS FILOSÓFICOS CONSOLIDADOS
Sostenibilidad > Intensidad
Claridad > Motivación
Tendencia > Momento
Protección no es fracaso
Expansión requiere base estable
El sistema decide el estado
El usuario conserva agencia en ejecución

Cómo interactúa el usuario.
Qué se mide.
Cuándo el sistema habla o calla.

Cómo evoluciona el proceso.
Cómo se mantiene equilibrio empatía/productividad.
2️⃣ Estructura General del Bloque
Incluye:
🎙 Bloque D – Input por audio
📊 Bloque E – Datos y memoria

3️⃣ INPUT POR AUDIO (Bloque D)
3.1 Principio central
El audio es libre.
No tiene:
estructura obligatoria
duración fija
guión
formato preestablecido
El usuario puede:
descargar
contar
pedir ayuda
compartir logros
expresar cansancio
ajustar proceso
3.2 El sistema siempre escucha en dos capas
Capa 1 – Medición (siempre activa)
Se detecta:
Energía
Carga emocional
Valencia (positiva / negativa / mixta)
Claridad vs confusión
Intención (descargar / buscar / compartir)
Relación con el hábito
Fase del proceso
Esto ocurre siempre, incluso en descarga.
Capa 2 – Intervención (condicional)
Depende del estado operativo.
En descarga:
Se mide todo.
No se sugiere nada.
No se estructura.
No se etiqueta.
Principio:
Se analiza siempre.
Se interviene solo cuando hay disponibilidad.
4️⃣ DATOS Y MEMORIA (Bloque E)
4.1 Qué se guarda
No se guardan audios crudos innecesarios.
Se guarda:
Patrones emocionales
Tendencias de energía
Consistencia
Fase del proceso
Relación con disciplina
Cambios de rigurosidad
Narrativa de identidad emergente
4.2 Baseline dinámico
Se construye un baseline personal que:
No es comparación externa
No es promedio estadístico simple
Es “cómo suele estar este usuario cuando está bien para sí mismo”
El baseline:
Se recalibra con el tiempo
Distingue estado puntual vs cambio real
4.3 Memoria narrativa
HumanHab recuerda procesos, no eventos aislados.
No:
“El 12 discutiste.”
Sí:
“Ante conflicto sostenido tiendes a aumentar exigencia.”
5️⃣ ESTADOS OPERATIVOS (NO emocionales)
Se definieron estados que determinan comportamiento del sistema:
Inicio
Sostén frágil
Disciplina emergente
Consolidación
Pausa consciente / desconexión
Estos estados no describen la persona.
Describen cómo el sistema debe comportarse.
6️⃣ FRECUENCIA POR FASE
La frecuencia depende de:
Intención explícita
Fase del proceso
Energía sostenida
Relación con disciplina
Principios:
No reaccionar a un solo día.
Reducir frecuencia es respeto, no abandono.
El silencio es parte del diseño.
7️⃣ EQUILIBRIO EMPATÍA + PRODUCTIVIDAD

Energía y disciplina son ejes distintos.
Ejemplo válido:
Energía baja
Dirección alta (disciplina consolidada)
Regla clave:
Baja energía no implica bajar rigurosidad automáticamente.
El sistema:
No infantiliza.
No protege en exceso.
No empuja en fractura real.
No frena identidad construida.
8️⃣ RIGUROSIDAD ADAPTATIVA
8.1 Qué es rigurosidad
Combinación de:
Nivel de exigencia
Tolerancia al desvío
Frecuencia de seguimiento
Grado de estructura
Nivel de desafío
8.2 Niveles definidos
🔹 Nivel 1 – Sostén mínimo
🔹 Nivel 2 – Proceso estable (default)
🔹 Nivel 3 – Expansión / Alto compromiso
No más niveles.
8.3 Regla central
La rigurosidad no cambia por emoción puntual.
Se ajusta por coherencia entre:
Energía sostenida
Fase del proceso
Identidad construida
Consistencia reciente
9️⃣ FLUJO DE AJUSTE DE RIGUROSIDAD
El usuario siempre inicia.
Nunca hay cambios automáticos invisibles.
Proceso:
Usuario solicita ajuste.
Define intención (subir / bajar / revisar).
Sistema evalúa coherencia.
Se da uno de tres resultados:
Aprobación directa
Aprobación con advertencia
Sugerencia alternativa (no bloqueo)
Si el usuario insiste, puede ejecutar.
Pero con conciencia explícita.
10️⃣ REGLAS FUERTES
No se agregan múltiples hábitos si el actual no está estable.
No se baja rigurosidad por cansancio puntual.
No se sube por euforia impulsiva.
No se bloquea autonomía.
No se ejecutan cambios sin validación.
11️⃣ Cambios frecuentes de nivel
Si hay ajustes repetidos en pocos días:
El sistema refleja inestabilidad decisional.
No acusa.
Invita a sostener antes de modificar.
12️⃣ Lo que NO es Bloque 3
No incluye aún:
Tono emocional profundo
Técnicas de regulación
Prevención de burnout
Límites de bienestar
Diseño detallado de mensajes

13️⃣ Principios estructurales consolidados
Medir siempre, intervenir con criterio.
Autonomía con validación.
Productividad sin violencia.
Empatía sin infantilización.
Identidad > cumplimiento.

Rigurosidad contextual.
Transparencia total en ajustes.

BLOQUE 4 — INTERVENCIÓN Y ACOMPAÑAMIENTO
Síntesis Estratégica y Decisiones Tomadas
1️⃣ Propósito del Bloque
Este bloque no define qué hace HumanHab.
Define:
Cómo se expresa.
Cuándo interviene.
Cómo regula.
Qué tono utiliza.
Qué límites tiene.
Qué nunca hace.
Hasta dónde llega.
Cómo maneja memoria, error y rigurosidad.
Es el bloque que define el carácter conductual del sistema.
2️⃣ Posicionamiento Conductual
HumanHab es:
Un sistema de calibración entre intención, energía y acción.
No es:
Coach motivacional.
App de frases.
Sistema frío de métricas.
Terapia.
Sustituto de ayuda profesional.
Se posiciona como:
Regulador cálido.
Presencia funcional.
Autoridad tranquila.
Sistema de entrenamiento progresivo.
3️⃣ Personalidad Definida
Se definieron tres ejes:
Confrontación → Baja
Sube ligeramente ante patrón consolidado.
Calidez → Alta (funcional, no emocional)
Sin dramatismo, sin validación excesiva.
Tecnicidad → Baja por defecto, media para usuarios avanzados.
4️⃣ Identidad Vocal
Decisiones clave:
No se nombra a sí mismo.
No habla en tercera persona.
No usa tono corporativo.
No usa frases motivacionales.
No utiliza emocionalismo artificial.
Puede usar primera persona funcional:
“Noté…”
“Detecté…”
“Podemos…”
Pero:
No usa lenguaje afectivo (“me preocupa”, “estoy orgulloso”).
No exagera presencia.
Se percibe como inteligencia integrada, no como entidad externa.
5️⃣ Umbral de Intervención
Decisión crítica:
HumanHab no interviene ante cada señal.
Modelo definido:
Señal leve → Registra, no comunica.
Tendencia emergente → Micro intervención.
Patrón consolidado → Intervención clara.
Riesgo real → Regulación activa (según rigor).
Silencio estratégico es parte del diseño.
6️⃣ Tipos de Intervención
Se definieron cuatro categorías:
Micro-calibración.
Corrección de trayectoria.
Regulación activa.
Feedback estratégico periódico.
Intervenciones:
Breves.
Directas.
Contextuales.
Sin sobreexplicación.
7️⃣ Rigurosidad
Se incorporó un sistema de rigor configurable.
Niveles:
Nivel 1: Sugiere.

Explicación disponible, no obligatoria.

Vigilancia demostrativa.

No es motivar.
Es sostener continuidad.

Nunca se cae a cero. Solo se reduce intensidad.

Nivel Reducido → versión comprimida.
Nivel Mínimo → microacción simbólica + registro.
Preservación de vínculo → check-in sin ruptura.
Esto elimina:
Reinicios.
Streaks rotos.
Sensación de fracaso binario.

Interviene antes de:
Caída prolongada.
Sobrecarga crítica.
Abandono psicológico.
No espera 3–5 días de fallo para actuar.
Previene mediante:
Lectura de energía.
Estrés.
Ritmo acumulado.
Uso repetido de esfuerzo extraordinario.

El índice existe, pero no se muestra.
Se calcula con:
Sueño.
Estrés percibido.
Intensidad de trabajo.
Uso de modo esfuerzo controlado.
Ritmo de recuperación.
Variabilidad reciente.
El usuario no ve un número.
Ve efectos:
Ajustes.
Simplificaciones.
Redistribución.
Regulación sugerida.
Motivo:
No generar presión ni obsesión métrica.

Energía baja

No imposición rígida.

No semana completa.

El foco es recuperación, no perfección.

Atraso.
“Volvé al plan”.
Permitido:
Ajustamos.
Simplificamos.

Este bloque cubrió:
Decisión de arquitectura.
Decisión sobre builders vs código propio.
Definición técnica base.
Estrategia de aprendizaje.
Preparación para ejecución real.
1️⃣ Decisiones Estratégicas Clave
✔ Se descarta usar Antigravity como núcleo
Motivo:
Mezcla UI y lógica.
Genera código acoplado.
Complica escalabilidad.
Riesgo alto en MVP2.
Se permite usarlo solo como:
Referencia visual.
Acelerador de componentes.
✔ Se elige React local como base
Motivos:
Control total de arquitectura.
Compatible con React Native.
Modular.
Escalable.
Profesional desde el inicio.
✔ Claude / IA se usan como copiloto
No como arquitecto.
Regla establecida:
Vos definís estructura.
IA escribe código dentro de esa estructura.
2️⃣ Arquitectura Frontend Definida
Estructura modular desde el día 1:
src/
  app/
  components/
  pages/
  features/
  services/
  lib/
  hooks/
Separación estricta:
UI Layer → solo renderiza.
Domain Layer (lib/) → algoritmo y reglas puras.
Data Layer (services/) → Firebase.
Nunca mezclar capas.
Esto evita reescritura futura.
3️⃣ Arquitectura Backend Definida
Se usa:
Firebase Auth
Firestore
Cloud Functions
Hosting
IA solo a través de Cloud Function.
Nunca directa desde frontend.
4️⃣ Compatibilidad futura con React Native
Decisión crítica:
La lógica portable (lib/ + services/) será reutilizable.
Al pasar a React Native:
Solo se reescribe UI.
No se reescribe lógica ni modelo de datos.
Esto elimina el miedo a rehacer todo.
5️⃣ Estrategia de Desarrollo Adoptada
Plan de ejecución semanal definido:
Día 1 — Setup React + Tailwind
Día 2 — Pantallas estáticas + algoritmo
Día 3 — Firebase Auth + Firestore
Día 4 — Lógica real + fase
Día 5 — IA + Deploy
Objetivo:
MVP1 funcional en una semana.
6️⃣ Aclaración Crítica Sobre Builders
Se estableció que el problema no es usar agentes.
El problema sería:
Delegar arquitectura.
HumanHab tiene:
Separación algoritmo/IA.
Roadmap MVP1 → MVP3.
Multi-ciclo backend.
Fase como centro regulador.
Sistema modular.
Eso no puede quedar en manos de un builder.
7️⃣ Miedos Detectados y Resueltos
Miedo identificado:
“No saber lo suficiente como para hacerlo funcionar.”
Respuesta estructural:
Solo necesitás dominar:
Componentes.
useState.
Router.
Guardar/leer Firestore.
Llamar Cloud Function.
Nada más.
El resto lo escribe IA.
8️⃣ Estado Final del Proyecto al Cierre del Bloque
✔ Visión definida
✔ MVP1 operativo definido
✔ Edge cases blindados
✔ Documento Operativo formalizado
✔ SDD formalizado
✔ Arquitectura modular definida
✔ Decisión tecnológica tomada
✔ Estrategia de ejecución clara
✔ Compatibilidad futura asegurada
